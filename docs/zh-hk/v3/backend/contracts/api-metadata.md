# 接口元數據契約

接口元數據契約用於保證不同後端實現生成一致的 API 文檔和接口描述。MineAdmin 推薦使用 OpenAPI/Swagger 作為公共表達形式，框架實現負責把自身註解、屬性或配置轉換為同一套元數據。

接口元數據不只服務於 Swagger UI。它同時影響前台類型生成、表單約束、權限按鈕、調試工具和操作日誌。因此，元數據應被視為“接口行為的一部分”，而不是代碼寫完後的附屬説明。

## 必填信息

每個後台接口至少應描述以下信息：

| 信息 | 説明 |
|------|------|
| 路徑與方法 | 接口路徑、HTTP 方法和資源語義。 |
| 分組標籤 | 所屬業務模塊，便於 Swagger UI 和前台工具識別。 |
| 認證要求 | 是否需要登錄、Token 類型和安全方案。 |
| 權限標識 | 後台菜單、按鈕或接口權限編碼。 |
| 請求參數 | 路徑參數、查詢參數、請求體和文件字段。 |
| 響應結構 | 成功、失敗、分頁和業務異常的響應模型。 |

## 元數據來源

不同框架可以使用註解、屬性、路由配置或獨立描述文件生成 OpenAPI，但最終語義需要對齊。以當前 Hyperf 實現為例，元數據主要來自以下位置：

| 來源 | 契約語義 |
|------|----------|
| HTTP 方法註解 | `path`、`operationId`、`summary`、`tags`、`security`。 |
| `Permission` 屬性 | 後台權限標識，應與菜單和按鈕權限一致。 |
| 表單請求對象 | 請求體字段、必填字段、驗證語義和字段説明。 |
| Schema 類 | 響應字段、字段類型、嵌套對象和數組元素類型。 |
| `ResultResponse` | 普通接口統一響應結構。 |
| `PageResponse` | 列表接口分頁響應結構。 |
| 全局 OpenAPI 配置 | 文檔版本、服務地址、認證方案和外部文檔入口。 |

如果某個實現無法把這些信息直接放進 OpenAPI 標準字段，應使用穩定的擴展字段承載 MineAdmin 語義，例如 `x-permission`、`x-request-source` 或等效字段。擴展字段名稱一旦公開，應按接口契約維護。

## 路由操作元數據

每個接口操作應保持以下字段穩定：

| 字段 | 要求 |
|------|------|
| `operationId` | 在同一服務內唯一，作為客户端生成、接口變更識別和調試定位的穩定鍵。 |
| `summary` | 使用清晰的業務動作名稱。當前實現也會用它記錄操作日誌，因此不要寫成“接口一”“處理數據”等空泛描述。 |
| `tags` | 對應後台功能模塊或資源歸屬，同一模塊的接口應歸入同一標籤。 |
| `path` | 與後台路由契約一致，路徑參數名稱需要和方法參數、文檔參數保持一致。 |
| `method` | 與真實 HTTP 方法一致，不能為了文檔分組而改變語義。 |
| `security` | 準確聲明接口是否需要訪問令牌、刷新令牌或無需認證。 |

認證、權限和審計依賴的接口不應省略操作元數據。對於會改變系統狀態的接口，`summary` 應能直接表達被記錄的業務動作。

## 認證與權限元數據

MineAdmin 後台接口默認使用統一令牌體系。OpenAPI 中應聲明全局安全方案，並在接口級別明確引用：

| 方案 | 説明 |
|------|------|
| `Bearer` | HTTP Bearer Token，承載 JWT 訪問令牌或刷新令牌。 |
| `ApiKey` | Header 中的 `token`，用於兼容需要額外令牌的場景。 |

接口是否需要認證應以實際中間件和路由語義為準。登錄接口不聲明認證要求；退出、當前用户信息、普通後台資源和上傳接口應聲明認證要求；刷新令牌接口應明確它使用刷新令牌語義。

權限標識來自後端授權規則，也被前台按鈕和菜單使用。接口元數據應保留權限編碼，例如 `permission:user:index`、`permission:user:save`、`dataCenter:attachment:upload`。當 OpenAPI 標準字段無法表達權限時，需要通過擴展字段或生成器配置保留該值。

## 請求元數據

請求描述應覆蓋參數位置、字段類型、必填狀態和驗證語義：

| 類型 | 要求 |
|------|------|
| 路徑參數 | 在 `path` 中出現的參數必須聲明為必填，並標註類型。 |
| 查詢參數 | 列表、搜索、排序和分頁參數應聲明為查詢參數或等效請求參數。 |
| JSON 請求體 | 創建、更新、授權等接口應通過 request body 描述字段。 |
| 文件上傳 | 使用 `multipart/form-data`，附件上傳字段名固定為 `file`。 |
| 批量參數 | ID 數組、權限數組、角色代碼數組等應標註數組元素類型。 |

表單請求對象中的驗證規則和文檔必填字段應保持一致。例如規則中 `required`、`required_with`、`sometimes` 對應的必填/條件必填語義，需要在請求 schema 或字段説明裏體現。字段白名單也需要一致，避免文檔展示了後端不會接收的字段。

## 響應元數據

後台接口響應應統一描述為 `Result` 包裝結構：

```json
{
  "code": 200,
  "message": "成功",
  "data": {}
}
```

`code` 使用業務狀態碼，`message` 為業務消息，`data` 承載真實數據。接口元數據應描述 `data` 的具體結構，而不是隻聲明為泛型對象。

常見響應類型如下：

| 類型 | `data` 結構 |
|------|-------------|
| 普通成功 | 對象、數組或空數組，按業務 schema 描述。 |
| 分頁列表 | 至少包含 `list` 和 `total`；`list` 元素類型必須指向對應 Schema。 |
| 登錄成功 | 包含 `access_token`、`refresh_token` 和 `expire_at`。 |
| 驗證失敗 | 使用統一錯誤響應，業務狀態碼為 `422`。 |
| 未認證/未授權 | 使用統一錯誤響應，業務狀態碼分別為 `401`、`403`。 |
| 禁用狀態 | 使用統一錯誤響應，業務狀態碼為 `423`。 |

`ResultCode` 的枚舉值應進入接口文檔，至少覆蓋成功、失敗、未認證、未授權、未找到、方法不允許、不可接受、驗證失敗和禁用等場景。不同框架的異常處理可以不同，但對外元數據中的業務狀態碼語義需要保持一致。

## Schema 約定

Schema 用於描述穩定的對外字段，不應直接暴露內部模型實現細節。

- Schema 名稱應穩定，推薦使用資源名加 `Schema` 的形式。
- 字段名以最終 JSON 字段為準，例如 `storage_mode`、`origin_name`，不要讓 PHP、JavaScript 或 ORM 的內部命名泄漏到契約中。
- 字段類型應使用 OpenAPI 可識別類型；數組字段需要聲明元素類型。
- 嵌套關係應引用對應 Schema，避免把複雜對象退化成無結構的 `array`。
- 時間、枚舉、狀態值和文件字段應補充説明或示例。
- 請求 Schema 和響應 Schema 可以複用基礎字段，但必須允許按場景過濾字段，避免創建接口、更新接口和返回對象混在一起。

## 分頁與通用能力

列表接口應使用統一分頁元數據：

| 項 | 約定 |
|----|------|
| 請求頁碼 | `page`，默認 `1`。 |
| 每頁數量 | `page_size`，默認 `10`。 |
| 列表數據 | `data.list`。 |
| 總數 | `data.total`。 |

導入、導出、上傳、批量刪除、批量授權等通用能力應使用固定的命名和結構。比如附件上傳的 request body 必須能看出 `file` 是文件字段，批量授權用户角色的 `role_codes` 必須標註為字符串數組。

## 元數據一致性

- 同一接口在不同框架實現中應生成一致的路徑、方法、標籤和響應模型。
- 字段名稱、類型、必填狀態和示例值應與真實接口保持一致。
- 權限標識應和菜單/按鈕配置共用同一套編碼。
- 分頁、導入導出、文件上傳等通用能力應使用統一描述方式。
- 操作名稱、權限編碼和響應 Schema 的變更應被視為兼容性變更。
- 對已公開字段進行重命名或刪除時，應先通過廢棄説明、版本説明或新增字段過渡。

## 示例

以下示例展示接口元數據應表達的核心語義，不限定具體框架寫法：

```yaml
paths:
  /admin/user/list:
    get:
      operationId: userList
      summary: 用户列表
      tags:
        - 用户管理
      security:
        - Bearer: []
          ApiKey: []
      x-permission: permission:user:index
      parameters:
        - name: page
          in: query
          schema:
            type: integer
            default: 1
        - name: page_size
          in: query
          schema:
            type: integer
            default: 10
      responses:
        "200":
          description: 成功
```

## 與前台模板的關係

前台模板可以基於接口元數據生成請求類型、表單約束或調試文檔。因此接口元數據不是單純的説明文檔，而是前後端協作的穩定契約。

前台依賴接口元數據時，通常會讀取以下信息：

- 根據 `operationId` 和路徑生成請求方法。
- 根據 request body 和參數 schema 推導表單字段與類型。
- 根據響應 schema 推導列表、詳情和選擇器的數據類型。
- 根據權限標識控制按鈕、菜單和頁面能力。
- 根據認證聲明決定是否附加訪問令牌或刷新令牌。
