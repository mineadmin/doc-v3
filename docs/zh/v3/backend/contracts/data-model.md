# 数据模型契约

数据模型契约定义 MineAdmin 后端实现之间共享的业务实体和关系。不同框架可以使用不同 ORM 或数据访问层，但模型的业务语义、关系边界和对外字段含义需要保持一致。

## 分层约定

MineAdmin 后端通常按以下职责拆分：

| 层级 | 职责 |
|------|------|
| Controller | 接收请求、触发验证、调用服务并返回统一响应。 |
| Service | 编排业务流程，处理跨模型、跨仓储的业务规则。 |
| Repository | 封装查询、持久化和复杂数据组装。 |
| Model | 描述数据库表、实体关系、类型转换和模型事件。 |

框架实现可以调整目录名称或基类，但不应改变这些职责边界。

## 核心实体

以下实体属于前台模板、权限系统和后台接口共同依赖的核心模型：

- 用户：后台账号、状态、认证凭据和用户画像。
- 角色：权限集合和用户授权关系。
- 菜单/权限：路由菜单、按钮权限、接口权限标识。
- 部门/岗位：组织结构和数据权限归属。
- 附件：上传文件、存储驱动、访问地址和创建人。
- 插件：插件元数据、启停状态和扩展能力。

新增框架实现时，应优先复用这些实体语义，避免为了适配框架而改变前台和插件感知到的数据结构。

## 关系约定

- 用户可以关联多个角色，并通过角色获得菜单和按钮权限。
- 用户可以关联部门、岗位或其他组织维度，用于数据权限过滤。
- 菜单既承载前台路由，也承载后台权限标识。
- 附件记录应能追踪上传者、存储位置、原始文件信息和访问 URL。
- 插件实体应能关联安装状态、配置入口和前后端扩展点。

## 扩展原则

新增字段、模型或关系时，应先判断是否会影响前台模板、接口元数据或响应结构。如果会影响，需要同步更新公共契约，再补充各框架实现细节。
