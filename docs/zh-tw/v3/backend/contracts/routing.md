# 後臺路由契約

後臺路由契約定義管理端資源的訪問路徑、HTTP 方法、許可權邊界和操作語義。不同框架的路由註冊方式可以不同，但對外暴露的後臺介面應保持一致。

本文依據 MineAdmin 當前管理端控制器整理，描述前臺模板、許可權系統、介面文件和審計日誌共同依賴的穩定約定。

## 基本原則

- 同一資源在不同實現中使用一致的路徑語義。
- 列表、詳情、建立、更新、刪除等常見操作保持穩定的 HTTP 方法和引數位置。
- 需要登入的後臺介面必須經過統一認證流程。
- 需要授權的後臺介面必須繫結穩定的許可權標識。
- 路由元資料應能支撐介面文件、前臺能力識別和操作日誌。
- 批次操作、匯入匯出、狀態切換等擴充套件操作應有清晰的資源歸屬。

## URL 名稱空間

後臺介面統一使用 `/admin` 作為管理端名稱空間，名稱空間後的第一段通常表示業務域或資源。

| 型別 | 路徑形態 | 示例 |
|------|----------|------|
| 認證介面 | `/admin/passport/{action}` | `/admin/passport/login`、`/admin/passport/refresh` |
| 當前使用者能力 | `/admin/permission/{action}` | `/admin/permission/menus`、`/admin/permission/roles` |
| 資源管理 | `/admin/{resource}` | `/admin/user`、`/admin/role`、`/admin/department` |
| 資源列表 | `/admin/{resource}/list` | `/admin/user/list`、`/admin/attachment/list` |
| 資源子能力 | `/admin/{resource}/{id}/{relation}` | `/admin/role/{id}/permissions` |
| 外掛後臺介面 | `/admin/plugin/{plugin}/{action}` | `/admin/plugin/store/index` |

框架實現可以使用註解、路由表、控制器字首或檔案路由註冊這些地址，但最終公開 URL 不應因為框架差異而變化。

## 資源操作

| 操作 | HTTP 方法與路徑 | 引數位置 | 說明 |
|------|-----------------|----------|------|
| 列表 | `GET /admin/{resource}/list` | 查詢引數或請求引數 | 支援分頁、篩選、排序和資料許可權過濾。 |
| 建立 | `POST /admin/{resource}` | JSON 請求體 | 請求體承載表單資料。 |
| 更新 | `PUT /admin/{resource}/{id}` | 路徑引數 + JSON 請求體 | 路徑引數定位資源，請求體承載可變更欄位。 |
| 刪除 | `DELETE /admin/{resource}` | 請求引數或請求體 | 當前多數管理資源透過請求資料傳入單個或多個 ID。 |
| 單條刪除 | `DELETE /admin/{resource}/{id}` | 路徑引數 | 適用於已經明確用路徑引數定位的資源，例如附件。 |
| 關係讀取 | `GET /admin/{resource}/{id}/{relation}` | 路徑引數 | 查詢使用者角色、角色許可權等資源關係。 |
| 關係寫入 | `PUT /admin/{resource}/{id}/{relation}` | 路徑引數 + JSON 請求體 | 批次授予角色、許可權或資料範圍。 |
| 資源動作 | `POST/PUT /admin/{resource}/{action}` | 依動作而定 | 上傳、重置密碼、狀態切換等非標準 CRUD 動作。 |

## 認證路由

認證路由屬於後臺契約的一部分，但不按普通資源許可權處理。

| 路徑 | 方法 | 認證要求 | 說明 |
|------|------|----------|------|
| `/admin/passport/login` | `POST` | 不需要訪問令牌 | 使用使用者名稱和密碼換取訪問令牌與重新整理令牌。 |
| `/admin/passport/logout` | `POST` | 需要訪問令牌 | 退出當前登入狀態。 |
| `/admin/passport/getInfo` | `GET` | 需要訪問令牌 | 返回當前登入使用者的基礎資訊和後臺設定。 |
| `/admin/passport/refresh` | `POST` | 需要重新整理令牌 | 換取新的訪問令牌。 |

## 當前使用者能力路由

當前使用者能力路由用於前臺初始化，不等同於普通資源管理介面。

| 路徑 | 方法 | 說明 |
|------|------|------|
| `/admin/permission/menus` | `GET` | 返回當前使用者可訪問的選單樹。 |
| `/admin/permission/roles` | `GET` | 返回當前使用者擁有的角色。 |
| `/admin/permission/update` | `POST` | 修改當前使用者資料或密碼。 |

這些介面必須保持響應欄位穩定，因為前臺模板會用它們完成路由生成、選單渲染和使用者資訊展示。

## 引數約定

- 分頁引數固定為 `page` 和 `page_size`，預設值分別為 `1` 和 `10`。
- 列表篩選欄位隨資源定義，但應透過請求引數統一傳入，並由資源倉儲或服務解釋。
- 建立和更新介面使用 JSON 請求體，欄位以對應表單請求或介面元資料為準。
- 檔案上傳使用 `multipart/form-data`，附件上傳欄位名固定為 `file`。
- 批次刪除介面通常使用 `ids` 或資源 ID 陣列；如果介面已經宣告路徑引數，應以路徑引數定位單個資源。
- 使用者授權角色使用 `role_codes` 陣列，角色授權許可權使用 `permissions` 陣列，陣列元素分別對應角色程式碼和選單許可權標識。
- 崗位資料許可權使用 `policy_type` 描述策略型別，`value` 描述策略值。

## 許可權標識

許可權標識應與後臺選單、按鈕和介面元資料保持一致。前臺模板根據許可權標識控制按鈕和頁面能力，後端實現根據同一標識執行授權校驗。

許可權標識推薦使用 `業務域:資源:動作` 的形式，例如：

| 場景 | 許可權標識示例 |
|------|--------------|
| 使用者列表 | `permission:user:index` |
| 建立使用者 | `permission:user:save` |
| 刪除使用者 | `permission:user:delete` |
| 授權使用者角色 | `permission:user:setRole` |
| 角色授權許可權 | `permission:role:setMenu` |
| 附件上傳 | `dataCenter:attachment:upload` |
| 登入日誌列表 | `log:userLogin:list` |

後端授權校驗應以該標識為準，前臺選單、按鈕許可權和介面元資料也應複用同一標識，避免同一能力出現多個命名。

## 路由元資料

每個後臺介面應提供穩定的路由元資料，至少覆蓋以下語義：

| 元資料 | 作用 |
|--------|------|
| `operationId` | 介面唯一操作名，用於介面文件、客戶端生成和變更識別。 |
| `summary` | 介面業務名稱，用於介面文件和操作日誌。 |
| `tags` | 介面分組，應與後臺功能模組或資源歸屬一致。 |
| `security` | 宣告介面需要訪問令牌、重新整理令牌或無需認證。 |
| 請求體 schema | 描述建立、更新、授權、上傳等介面的輸入欄位。 |
| 響應 schema | 描述列表、詳情、當前使用者能力等介面的返回結構。 |

如果某個框架沒有原生註解能力，也需要在路由註冊、OpenAPI 生成或等效配置中儲存這些語義。

## 審計約定

需要記錄操作日誌的後臺介面，應保留可識別的 HTTP 方法、真實路徑和業務名稱。操作日誌至少需要能夠追蹤：

- 操作使用者。
- 請求方法。
- 請求路徑。
- 業務名稱。
- 客戶端 IP。

因此，涉及新增、修改、刪除、授權、上傳等會改變系統狀態的介面，應確保路由元資料中的業務名稱明確，不使用空泛描述。

## 外掛路由

外掛可以註冊自己的後臺路由字首，但仍應遵守管理端路由契約：

- 路徑歸入 `/admin/plugin/{plugin}` 或外掛明確宣告的後臺名稱空間。
- 需要登入的外掛介面必須使用統一認證流程。
- 會改變系統狀態的外掛介面應使用合適的 HTTP 方法，並提供可審計的業務名稱。
- 外掛介面返回結構應保持與後臺統一響應契約一致。

## 框架實現

Hyperf 當前透過註解、控制器和中介軟體承載路由與 API 文件生成，具體用法見 [Hyperf 路由與 API 文件](/backend/frameworks/hyperf/3.2/base/router)。
