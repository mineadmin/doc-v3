# 介面元資料契約

介面元資料契約用於保證不同後端實現生成一致的 API 文件和介面描述。MineAdmin 推薦使用 OpenAPI/Swagger 作為公共表達形式，框架實現負責把自身註解、屬性或配置轉換為同一套元資料。

介面元資料不只服務於 Swagger UI。它同時影響前臺型別生成、表單約束、許可權按鈕、除錯工具和操作日誌。因此，元資料應被視為“介面行為的一部分”，而不是程式碼寫完後的附屬說明。

## 必填資訊

每個後臺介面至少應描述以下資訊：

| 資訊 | 說明 |
|------|------|
| 路徑與方法 | 介面路徑、HTTP 方法和資源語義。 |
| 分組標籤 | 所屬業務模組，便於 Swagger UI 和前臺工具識別。 |
| 認證要求 | 是否需要登入、Token 型別和安全方案。 |
| 許可權標識 | 後臺選單、按鈕或介面許可權編碼。 |
| 請求引數 | 路徑引數、查詢引數、請求體和檔案欄位。 |
| 響應結構 | 成功、失敗、分頁和業務異常的響應模型。 |

## 元資料來源

不同框架可以使用註解、屬性、路由配置或獨立描述檔案生成 OpenAPI，但最終語義需要對齊。以當前 Hyperf 實現為例，元資料主要來自以下位置：

| 來源 | 契約語義 |
|------|----------|
| HTTP 方法註解 | `path`、`operationId`、`summary`、`tags`、`security`。 |
| `Permission` 屬性 | 後臺許可權標識，應與選單和按鈕許可權一致。 |
| 表單請求物件 | 請求體欄位、必填欄位、驗證語義和欄位說明。 |
| Schema 類 | 響應欄位、欄位型別、巢狀物件和陣列元素型別。 |
| `ResultResponse` | 普通介面統一響應結構。 |
| `PageResponse` | 列表介面分頁響應結構。 |
| 全域性 OpenAPI 配置 | 文件版本、服務地址、認證方案和外部文件入口。 |

如果某個實現無法把這些資訊直接放進 OpenAPI 標準欄位，應使用穩定的擴充套件欄位承載 MineAdmin 語義，例如 `x-permission`、`x-request-source` 或等效欄位。擴充套件欄位名稱一旦公開，應按介面契約維護。

## 路由操作元資料

每個介面操作應保持以下欄位穩定：

| 欄位 | 要求 |
|------|------|
| `operationId` | 在同一服務內唯一，作為客戶端生成、介面變更識別和除錯定位的穩定鍵。 |
| `summary` | 使用清晰的業務動作名稱。當前實現也會用它記錄操作日誌，因此不要寫成“介面一”“處理資料”等空泛描述。 |
| `tags` | 對應後臺功能模組或資源歸屬，同一模組的介面應歸入同一標籤。 |
| `path` | 與後臺路由契約一致，路徑引數名稱需要和方法引數、文件引數保持一致。 |
| `method` | 與真實 HTTP 方法一致，不能為了文件分組而改變語義。 |
| `security` | 準確宣告介面是否需要訪問令牌、重新整理令牌或無需認證。 |

認證、許可權和審計依賴的介面不應省略操作元資料。對於會改變系統狀態的介面，`summary` 應能直接表達被記錄的業務動作。

## 認證與許可權元資料

MineAdmin 後臺介面預設使用統一令牌體系。OpenAPI 中應宣告全域性安全方案，並在介面級別明確引用：

| 方案 | 說明 |
|------|------|
| `Bearer` | HTTP Bearer Token，承載 JWT 訪問令牌或重新整理令牌。 |
| `ApiKey` | Header 中的 `token`，用於相容需要額外令牌的場景。 |

介面是否需要認證應以實際中介軟體和路由語義為準。登入介面不宣告認證要求；退出、當前使用者資訊、普通後臺資源和上傳介面應宣告認證要求；重新整理令牌介面應明確它使用重新整理令牌語義。

許可權標識來自後端授權規則，也被前臺按鈕和選單使用。介面元資料應保留許可權編碼，例如 `permission:user:index`、`permission:user:save`、`dataCenter:attachment:upload`。當 OpenAPI 標準欄位無法表達許可權時，需要透過擴充套件欄位或生成器配置保留該值。

## 請求元資料

請求描述應覆蓋引數位置、欄位型別、必填狀態和驗證語義：

| 型別 | 要求 |
|------|------|
| 路徑引數 | 在 `path` 中出現的引數必須宣告為必填，並標註型別。 |
| 查詢引數 | 列表、搜尋、排序和分頁引數應宣告為查詢引數或等效請求引數。 |
| JSON 請求體 | 建立、更新、授權等介面應透過 request body 描述欄位。 |
| 檔案上傳 | 使用 `multipart/form-data`，附件上傳欄位名固定為 `file`。 |
| 批次引數 | ID 陣列、許可權陣列、角色程式碼陣列等應標註陣列元素型別。 |

表單請求物件中的驗證規則和文件必填欄位應保持一致。例如規則中 `required`、`required_with`、`sometimes` 對應的必填/條件必填語義，需要在請求 schema 或欄位說明裡體現。欄位白名單也需要一致，避免文件展示了後端不會接收的欄位。

## 響應元資料

後臺介面響應應統一描述為 `Result` 包裝結構：

```json
{
  "code": 200,
  "message": "成功",
  "data": {}
}
```

`code` 使用業務狀態碼，`message` 為業務訊息，`data` 承載真實資料。介面元資料應描述 `data` 的具體結構，而不是隻宣告為泛型物件。

常見響應型別如下：

| 型別 | `data` 結構 |
|------|-------------|
| 普通成功 | 物件、陣列或空陣列，按業務 schema 描述。 |
| 分頁列表 | 至少包含 `list` 和 `total`；`list` 元素型別必須指向對應 Schema。 |
| 登入成功 | 包含 `access_token`、`refresh_token` 和 `expire_at`。 |
| 驗證失敗 | 使用統一錯誤響應，業務狀態碼為 `422`。 |
| 未認證/未授權 | 使用統一錯誤響應，業務狀態碼分別為 `401`、`403`。 |
| 停用狀態 | 使用統一錯誤響應，業務狀態碼為 `423`。 |

`ResultCode` 的列舉值應進入介面文件，至少覆蓋成功、失敗、未認證、未授權、未找到、方法不允許、不可接受、驗證失敗和停用等場景。不同框架的異常處理可以不同，但對外元資料中的業務狀態碼語義需要保持一致。

## Schema 約定

Schema 用於描述穩定的對外欄位，不應直接暴露內部模型實現細節。

- Schema 名稱應穩定，推薦使用資源名加 `Schema` 的形式。
- 欄位名以最終 JSON 欄位為準，例如 `storage_mode`、`origin_name`，不要讓 PHP、JavaScript 或 ORM 的內部命名洩漏到契約中。
- 欄位型別應使用 OpenAPI 可識別型別；陣列欄位需要宣告元素型別。
- 巢狀關係應引用對應 Schema，避免把複雜物件退化成無結構的 `array`。
- 時間、列舉、狀態值和檔案欄位應補充說明或示例。
- 請求 Schema 和響應 Schema 可以複用基礎欄位，但必須允許按場景過濾欄位，避免建立介面、更新介面和返回物件混在一起。

## 分頁與通用能力

列表介面應使用統一分頁元資料：

| 項 | 約定 |
|----|------|
| 請求頁碼 | `page`，預設 `1`。 |
| 每頁數量 | `page_size`，預設 `10`。 |
| 列表資料 | `data.list`。 |
| 總數 | `data.total`。 |

匯入、匯出、上傳、批次刪除、批次授權等通用能力應使用固定的命名和結構。比如附件上傳的 request body 必須能看出 `file` 是檔案欄位，批次授權使用者角色的 `role_codes` 必須標註為字串陣列。

## 元資料一致性

- 同一介面在不同框架實現中應生成一致的路徑、方法、標籤和響應模型。
- 欄位名稱、型別、必填狀態和示例值應與真實介面保持一致。
- 許可權標識應和選單/按鈕配置共用同一套編碼。
- 分頁、匯入匯出、檔案上傳等通用能力應使用統一描述方式。
- 操作名稱、許可權編碼和響應 Schema 的變更應被視為相容性變更。
- 對已公開欄位進行重新命名或刪除時，應先透過廢棄說明、版本說明或新增欄位過渡。

## 示例

以下示例展示介面元資料應表達的核心語義，不限定具體框架寫法：

```yaml
paths:
  /admin/user/list:
    get:
      operationId: userList
      summary: 使用者列表
      tags:
        - 使用者管理
      security:
        - Bearer: []
          ApiKey: []
      x-permission: permission:user:index
      parameters:
        - name: page
          in: query
          schema:
            type: integer
            default: 1
        - name: page_size
          in: query
          schema:
            type: integer
            default: 10
      responses:
        "200":
          description: 成功
```

## 與前臺模板的關係

前臺模板可以基於介面元資料生成請求型別、表單約束或除錯文件。因此介面元資料不是單純的說明文件，而是前後端協作的穩定契約。

前臺依賴介面元資料時，通常會讀取以下資訊：

- 根據 `operationId` 和路徑生成請求方法。
- 根據 request body 和引數 schema 推導表單欄位與型別。
- 根據響應 schema 推導列表、詳情和選擇器的資料型別。
- 根據許可權標識控制按鈕、選單和頁面能力。
- 根據認證宣告決定是否附加訪問令牌或重新整理令牌。
