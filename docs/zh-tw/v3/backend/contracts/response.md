# 響應結構契約

響應結構契約定義 MineAdmin API 的統一返回格式。不同框架實現可以使用不同的響應物件或異常處理器，但對外返回的資料結構需要保持一致。

## 基礎結構

標準響應包含三個核心欄位：

```json
{
  "code": 200,
  "message": "操作成功",
  "data": {}
}
```

| 欄位 | 說明 |
|------|------|
| `code` | 業務狀態碼，用於表達業務處理結果。 |
| `message` | 面向使用者或開發者的結果說明。 |
| `data` | 響應資料，可以是物件、陣列、分頁結構或空物件。 |

HTTP 狀態碼用於表達傳輸層結果，`code` 用於表達業務層結果。前臺模板應優先根據業務狀態碼和約定的異常處理流程展示提示。

## 分頁結構

分頁介面應在 `data` 中返回列表和分頁資訊，欄位命名應在同一產品版本內保持穩定。不同框架實現可以在內部使用不同分頁器，但不能改變前臺依賴的最終結構。

## 錯誤結構

業務錯誤、驗證錯誤、未認證、未授權和系統錯誤都應進入統一響應結構。框架實現需要把自身異常轉換為穩定的業務響應，避免直接暴露底層框架異常格式。

## 多語言訊息

業務訊息可以由後端實現根據客戶端語言生成，但訊息鍵、業務狀態碼和錯誤語義應保持一致。框架具體多語言載入方式見對應實現文件。

## 框架實現

Hyperf 當前透過 `Result`、`ResultCode` 和異常處理器統一響應結構，具體用法見 [Hyperf 錯誤處理](/backend/frameworks/hyperf/3.2/base/error-handler)。
